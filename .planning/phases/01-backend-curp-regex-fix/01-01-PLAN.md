<objective>
Fix the `CURP_REGEX` position-17 bug in `server/src/validation/schemas.js` so patients born in 2000 or later (who carry an alphabetic century differentiator in position 17) pass backend validation, and add regression tests in `server/tests/curp.test.js` so the bug cannot silently return.

Purpose: Post-2000 patients are currently rejected at backend validation — a blocking bug that prevents them from completing pre-registration and receiving a folio.
Output: One-line regex change plus 2-3 new `test()` blocks covering the post-2000 valid case and its corresponding invalid variants.
</objective>

## Phase Goal (MVP user story)

**As a** patient born in 2000 or later, **I want to** enter my CURP with a letter in position 17 and have it accepted during pre-registration, **so that** I can complete my registration and receive a valid folio instead of being wrongly rejected.

## Artifacts this phase produces

This phase creates **no new exported symbols**. It:
- Modifies the existing exported constant `CURP_REGEX` in `server/src/validation/schemas.js` (final two regex atoms only).
- Adds new `test()` blocks to `server/tests/curp.test.js`:
  - `acepta CURP post-2000 (letra en posición 17)`
  - `rechaza CURP con diferenciador o dígito verificador inválido`
- No new files, no new functions, no new API surface, no data-structure or API-contract changes.

## Specless probe — reviewed-and-inapplicable edge items (no silent drops)

Two auto-generated boundary probes for CURP-02 were reviewed and found not to apply to fixed-format 18-character string validation:
- **adjacency** ("when two things are exactly equal or just touch, do they merge/collide/separate?") — designed for collection/comparison logic; there is no comparison or merge semantics in a single-string regex match. Not applicable; not added to must_haves.
- **ordering** ("when elements compare equal, is output order specified and stable?") — designed for collection output ordering; regex validation returns a deterministic boolean with no output ordering. Not applicable; not added to must_haves.

The two applicable probes (**unclassified** → covered by the post-2000 acceptance truth; **empty/null** → covered by the existing required-field truth) are represented in `must_haves.truths` above.
