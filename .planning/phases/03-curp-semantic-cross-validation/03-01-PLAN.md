<objective>
Add a single cross-field validation refinement to `esquemaDatosPersonales` in
`server/src/validation/schemas.js` so the backend rejects any `datos_personales` payload whose
CURP-encoded birthdate (positions 5-10, `yymmdd`) or sex letter (position 11, `H`/`M`)
contradicts the separately captured `fechaNacimiento`/`sexo` fields, and lock the behavior with
backend `node:test` cases (CURP-07).

Purpose: A folio must never be issued against internally inconsistent identity data. This closes
the last gap where the format-valid CURP and the typed birthdate/sex could silently disagree.
Output: A `.superRefine()` block chained after `.strict()` on `esquemaDatosPersonales`, a fixed
pre-existing fixture, and four new semantic test cases — all inside the two files already scoped
by the phase; no change to `Fase`/`Draft`/`Record` structures or API contracts.
</objective>
