# Phase 2 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

## Scope Note (read first)

This phase's UI surface is narrow and already-styled — no new component, no new page, no new
interaction pattern:

1. **CURP masking fix** — `enmascararCurp` (`client/src/utils/mask.ts:3-6`) changes from exposing
   5 characters to 4. The only render site is `Confirmacion.tsx`'s `<dd>` in the `.resumen__fila`
   summary row (`client/src/components/wizard/phases/Confirmacion.tsx:69`). This is a **text-content
   change only** — the string returned is shorter by one visible character and one more `*`; no
   layout, spacing, color, or typography changes.
2. **CURP pattern tightening** — the `curp` field's `pattern` in `phases.config.json` becomes
   stricter (matches the corrected backend `CURP_REGEX`). This means the **existing**, unchanged
   error-message rendering path in `PhaseRenderer.tsx`'s `validarCampo` (line 11-21) and
   `Field.tsx`'s `.campo__error` (line 62) will now fire for more malformed inputs than before —
   the message text, style, and DOM position are unchanged; only the *trigger frequency* changes.

Per the phase's own scope note and `.claude/CLAUDE.md`'s "alcance estricto" directive, this contract
**confirms existing values as frozen**, rather than declaring new spacing/typography/color decisions.
No shadcn initialization was proposed — this project has no `components.json`, uses hand-written CSS
(BEM convention, `client/src/styles/styles.css`), and introducing a component-library dependency for
a two-string bug fix would itself be scope creep. This is a deliberate discretion call, not an
oversight — flagged here for the checker/executor.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Unchanged — `"Guardar y continuar"` (`PhaseRenderer.tsx:119`, phase submit) / `"Enviar registro"` (`Confirmacion.tsx:98`, final submit). No new CTA introduced. |
| Empty state heading | Not applicable — no list/collection UI in this phase's scope. |
| Empty state body | Not applicable. |
| Error state | Reused verbatim, no new copy: `` `${campo.etiqueta} no tiene un formato válido` `` (`PhaseRenderer.tsx:15`). For the `curp` field this renders as **"CURP no tiene un formato válido"**. The tightened pattern (CURP-05) causes this exact, pre-existing message to fire for input shapes that previously passed (e.g. malformed check digit, invalid entidad code). No wording change required. |
| Destructive confirmation | Not applicable — no destructive action in this phase's scope. |

**Behavior flag (not a copy change, but must be verified):** per `02-RESEARCH.md` Assumption A2,
`validarCampo` does not uppercase input before testing the pattern, while the backend does. A
lowercase-but-otherwise-valid CURP will now show the same **"CURP no tiene un formato válido"**
message client-side even though the backend would accept it after its own uppercase transform. This
is an accepted, in-scope consequence of CURP-05 (not a defect to silently patch) — the existing
`ayuda` copy already says "tal como aparece en tu identificación oficial," and CURPs are
conventionally presented uppercase on official IDs. Verification should explicitly exercise a
lowercase CURP against the new pattern to confirm this is the intended behavior, not an oversight.
